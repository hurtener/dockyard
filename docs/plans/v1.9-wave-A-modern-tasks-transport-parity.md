## Summary

The v1.9.1 bump to go-sdk `v1.7.0-pre.3` adopted SEP-2575 — a `2026-07-28`
server identifies itself in each result's `_meta` under
`io.modelcontextprotocol/serverInfo` — via `protocolcodec.EncodeServerInfo`,
injected by `responseSemanticsMiddleware` (D-199). Adversarial review of that
commit found the injection sits on the SDK server path, but the **Tasks Mount**
intercepts `tasks/*` frames *before* the SDK with an engine codec pinned to
`protocolcodec.DefaultVersion` (`2025-11-25`). This wave locks the invariant
that every modern-protocol Tasks response carries `serverInfo` **and**
`resultType`, hardens the stdio Mount against silently answering a
modern-tagged frame with a legacy shape, corrects the over-broad v1.9.1
CHANGELOG wording, and ships the regression tests that were missing.

## RFC anchor

- RFC §5.4 — the `protocolcodec` isolation seam (versioned codecs keyed on the
  negotiated `protocolVersion`).
- RFC §8.2 — the V1 Tasks implementation (the transport mount / shim).
- RFC §8.3 — Tasks lifecycle and methods.

## Briefs informing this phase

- brief 02 — MCP Tasks extension.
- brief 07 — MCP 2026-07-28 migration.

## Brief findings incorporated

- brief 07 §… — the `2026-07-28` protocol is negotiated per-request via
  `_meta` and `server/discover`, and (per the pinned SDK) is served on the
  streamable-HTTP transport **only when stateless**. This wave treats
  "modern protocol ⇒ HTTP-stateless" as the reachability model to prove, not
  assume.
- brief 02 §… — Tasks is a versioned extension whose wire envelopes must live
  behind `protocolcodec`; a Tasks response shape is a function of the
  negotiated protocol version, never hard-coded to one version.

## Findings I'm departing from (if any)

None. This wave reinforces D-199 and the RFC §5.4 rule (wire shape follows the
negotiated version) rather than departing from anything. Any change to the
Mount's version behaviour is recorded as a new decision (see Acceptance).

## Goals

- Prove and document the reachability model: which transports/modes can
  negotiate `2026-07-28`, and therefore where a modern Tasks response can
  legitimately occur.
- Guarantee that **every** modern-protocol Tasks response
  (`tasks/get`, `tasks/update`, `tasks/cancel`, `tasks/list`, `tasks/result`)
  carries both `serverInfo` (`_meta`, SEP-2575) and `resultType`, matching a
  non-Tasks modern response byte-for-byte on those two fields.
- Ensure the Mount never *silently* answers a modern-version-tagged frame with
  a legacy-shaped response: it either serves the correct modern shape or fails
  loudly with a typed error.
- Correct the v1.9.1 CHANGELOG claim that overstates coverage
  ("on every modern-protocol response").

## Non-goals

- Adding a modern-protocol negotiation path to the **stdio** transport. If the
  investigation confirms stdio is legacy-only, full modern-Tasks-over-stdio
  support is explicitly deferred (it would require threading a negotiated
  version and the server `Info` into the engine/Mount) and recorded as a V2
  backlog item, not built here.
- Any change to the `2026-07-28` *non-Tasks* response path — v1.9.1 already
  covers it and the review confirmed it correct.
- Advancing Dockyard's `2026-07-28` finalization claim (the SDK is still a
  prerelease — see `docs/specifications/README.md`).

## Public API surface

No new public surface is anticipated. If the investigation requires the Mount to
learn the negotiated version, the change is internal to `runtime/tasks`
(`Mount`/`Engine`) and `runtime/server`; handler-facing and manifest-facing APIs
are unchanged (P3). Any new exported symbol is added with a smoke assertion in
the same PR.

## Test plan

- **Unit:** `protocolcodec` already covers `EncodeServerInfo`; add unit coverage
  for any new Mount version-selection/guard branch (modern-tagged frame → modern
  shape or typed error; legacy frame → legacy shape).
- **Integration:** a real-driver end-to-end test standing up a stateless-HTTP
  `2026-07-28` server with a Tasks engine and asserting `serverInfo` +
  `resultType` on `tasks/get`/`update`/`cancel`/`list` (AGENTS.md §17 — this
  consumes the `runtime/tasks` ↔ `runtime/server` seam).
- **Concurrency / golden:** run the new paths under `-race`. If a Tasks wire
  envelope shape changes, re-pin the affected `protocolcodec` golden.

## Risks / open questions

- **Primary open question (drives scope):** is `2026-07-28` reachable over
  stdio at all? The pinned SDK gates the new protocol to stateless HTTP and
  Dockyard's `stdio.go` has no modern-protocol negotiation — strongly implying
  stdio is legacy-only and today's legacy-shaped stdio Tasks responses are
  *correct*. If so, this wave is document-the-invariant + guard + regression
  tests (a genuine quick release). If stdio can somehow carry a modern-tagged
  frame, the Mount must become version-aware, which enlarges the change.
- Making the Mount fully version-aware would need the engine to inject
  `serverInfo`, which requires plumbing the server `Info` into `runtime/tasks`
  — deliberately a **non-goal** here unless the investigation proves modern
  Tasks are reachable off the SDK path.

## Glossary additions

- **Transport parity** (if introduced) — the invariant that a response's wire
  shape depends only on the negotiated `protocolVersion`, not on which
  transport or internal dispatch path produced it. Add to `docs/glossary.md`
  in the implementing PR if the term is used in shipped prose.
