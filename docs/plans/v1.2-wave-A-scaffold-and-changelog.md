## Summary

Wave A closes two recorded post-V1 deferrals from `docs/V2-BACKLOG.md`,
both about "a developer lands at the right state on the first try":

1. **D-166 — scaffold auto-generate (supersedes D-139).** `dockyard new`
   (blank or `--template`) now runs `go mod tidy` and `dockyard generate`
   for the developer **at scaffold time**, so a fresh project reaches a
   green `dockyard validate` with no manual post-scaffold commands. The
   two steps run at the CLI boundary (`internal/cli/new.go`) — *not*
   inside `scaffold.Generate()` — so the pure scaffold package and its
   golden tests stay byte-identical. The steps are best-effort with a
   `--no-postgen` opt-out: a failed `go mod tidy` (e.g. offline / no
   module-proxy reach) prints a warning + the manual fallback and does
   **not** fail the scaffold. Closes the D-139 knock-on; the two extra
   steps drop out of the `scaffold-a-server` skill and the getting-started
   docs in the same PR (§19 hygiene).

2. **D-167 — Conventional-Commits changelog supplement.** The release
   pipeline's GitHub Release body stays the hand-authored CHANGELOG
   section (the canonical P1–P4 narrative — D-154), and now an
   auto-generated, Conventional-Commits-derived PR list is **appended
   under it**. The supplement is a pure, stdlib-only classifier in
   `internal/changelogx` (the same in-tree, golden-tested philosophy as
   D-157 — no third-party CommonMark / changelog-gen dependency on a
   load-bearing release gate); the `git log` invocation lives in the
   `cmd/changelogx` driver. The release workflow appends the block on a
   real tag push only (a dry-run keeps the bare extracted section). This
   is the V2 follow-up D-154 explicitly named.

Neither item changes a public runtime API or the `obs/v1` shape; both are
additive (a new opt-out flag; new generated release-body content). Under
the D-159 semver policy this is a **minor** (v1.2.0).

## Goals

1. `dockyard new <name>` (blank) and `dockyard new <name> --template
   <t>` both land at a green `dockyard validate` with **no** manual
   post-scaffold commands — the developer's next step is to write a
   handler, not to run plumbing.
2. The scaffold's post-step is robust offline: a `go mod tidy` that
   cannot reach the module proxy prints a clear warning + the exact
   manual fallback command and the scaffold still succeeds (exit 0, files
   written). `--no-postgen` skips both steps entirely for hermetic / CI /
   air-gapped runs.
3. The pure `scaffold` package and its golden tests are unchanged —
   `scaffold.Generate()` still returns the same byte-identical tree; the
   side-effecting steps live in `internal/cli/new.go`.
4. The published `scaffold-a-server` skill and the getting-started docs
   drop the two now-unnecessary post-scaffold steps (D-139 knock-on, §19).
5. A real tag-push release's GitHub Release body is the hand-authored
   CHANGELOG section (unchanged, canonical) **followed by** an
   auto-generated Conventional-Commits PR list grouped into Keep-a-
   Changelog categories.
6. The supplement is a pure, golden-tested, stdlib-only function (D-157
   philosophy) — a future authoring/format change is a unit-test failure
   in PR, never a release-time failure at tag push.

## Non-goals

- **Changing the canonical narrative.** Item 2 never rewrites or replaces
  the hand-authored CHANGELOG section; it only appends a supplement below
  it (D-154 — the hand-authored prose stays canonical).
- **Auto-generating the CHANGELOG file itself.** The `## [<version>]`
  section in `CHANGELOG.md` is still hand-authored before a release; the
  supplement is computed at release time and lives only in the GitHub
  Release body, not committed back to `CHANGELOG.md`.
- **An in-process `go mod tidy`.** Item 1 shells out to the `go`
  toolchain (the established pattern across the integration tests); it
  does not import `cmd/go` internals.
- **Running the post-steps inside `scaffold.Generate()`.** They live at
  the CLI boundary so the scaffold package stays a pure, network-free,
  golden-tested file generator.
- **A signing / provenance change** (that is a separate V2-backlog item).
- **Touching the `--dockyard-path` hidden pre-publish seam's semantics**
  (D-080) beyond letting `go mod tidy` resolve through whatever `go.mod`
  the scaffold already writes.

## Public API surface

- `dockyard new --no-postgen` — new boolean opt-out flag (additive;
  default false → post-steps run). The default behaviour change (now runs
  `go mod tidy` + `generate`) is additive from the developer's view: a
  superset of today's output, no removed capability.
- `internal/changelogx.Supplement([]Commit) string` — new pure function
  in an internal package. `internal/changelogx.Commit` — a new exported
  struct (`Type`, `Scope`, `Subject`, `PR`) the parser fills.
- `cmd/changelogx -supplement -from <ref> -to <ref> [-out <path>] [-append]`
  — new CLI mode on the existing driver.

No `runtime/*` public-API change; no manifest field; no `obs/v1` change.

## Risks / open questions

- **Risk: `go mod tidy` needs network the developer may not have.** The
  V2-backlog DoD says "without extra commands"; the offline reality is
  handled by making the step best-effort with a clear fallback message
  and a `--no-postgen` opt-out, so the scaffold never hard-fails on a
  network condition. Recorded in D-166 as a deliberate refinement of the
  DoD.
- **Risk: the post-step slows `dockyard new`.** `go mod tidy` on a fresh
  module is a few seconds against a warm proxy/cache; acceptable for a
  one-time scaffold. `--no-postgen` is the escape hatch for a tight loop.
- **Risk: the CC supplement misclassifies a non-Conventional commit.** A
  subject with no recognised prefix folds into `### Changed` (a safe
  catch-all); the hand-authored prose remains the canonical narrative, so
  a misclassified line in the supplement is cosmetic, not load-bearing.
- **Open question: should `docs`/`chore`/`test` commits appear at all?**
  Default: fold `docs`/`chore`/`test`/`ci`/`build` out of the supplement
  (they are noise in a release body); `feat`/`fix`/`perf`/`refactor` are
  the signal. Recorded under D-167 as a scope choice; trivially tunable in
  the pure function.

## Glossary additions

- "Scaffold post-step" — the `dockyard new` CLI-boundary phase that runs
  `go mod tidy` + `dockyard generate` after the pure scaffold writes its
  tree, so a fresh project is green under `dockyard validate` on the first
  try. Opt out with `--no-postgen`. Supersedes the D-139 manual workflow.
- "Changelog supplement" — the auto-generated, Conventional-Commits-
  derived PR list the release pipeline appends below the hand-authored
  CHANGELOG section in a GitHub Release body. The hand-authored prose
  stays canonical (D-154); the supplement is rendered by a pure,
  golden-tested `internal/changelogx.Supplement`.
