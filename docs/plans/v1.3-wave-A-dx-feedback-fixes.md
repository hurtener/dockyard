## Summary

Three code fixes from the first downstream MCP-Apps builder's feedback:

1. **D-168 — enforce the dead `require_fixtures` / `require_contract_tests`
   gates.** Both manifest flags exist and the blank scaffold sets them
   `true`, but no `validate` / `test` consumer reads them — they never
   bite. Wire them into the quality gates (the `attach-a-ui-resource` skill
   already documents the intended failure). The headline correctness fix.

2. **Scaffold `go.mod` version pin (the `v0.0.0` + `replace` sharp edge).**
   The scaffold writes `require github.com/hurtener/dockyard v0.0.0` (a
   placeholder behind the `replace`). Once a developer drops the `replace`
   to use the published module, `go get …@v1.2.0` fails with
   `v0.0.0: unknown revision` — they must hand-edit the `require`. Now that
   the Go module is published, pin a real version so the no-`replace` path
   upgrades cleanly.

3. **`dockyard new` empty-dir error message + `--here`.** `checkTarget`
   already allows a truly-empty existing dir; the friction was a dir with a
   hidden entry (`.git/`, `.gitignore`) rejected with a message that does
   not say *what* made it non-empty. Name the offending entries, and add an
   opt-in `--here` for "scaffold into this dir even though it has content"
   (the workflow the user hand-rolled with a temp dir + `mv`).

## Goals

1. A project with `require_fixtures: true` / `require_contract_tests: true`
   fails `dockyard validate` (Blocker) and `dockyard test` when a declared
   tool lacks its fixtures / contract test — matching what the
   `attach-a-ui-resource` skill already documents.
2. The default scaffold (which ships its fixtures + `greet_test.go`) stays
   green; only a project that removed them newly fails.
3. A scaffolded project whose `replace` is removed can `go get
   github.com/hurtener/dockyard@<published-version>` and build without a
   hand edit to `require`.
4. `dockyard new` into a dir with content reports **which entries** blocked
   it; `--here` scaffolds into a non-empty dir on purpose (no overwrite of
   colliding files — collisions are still a typed error).

## Non-goals

- Changing the four-state page rule or the fixture/contract-test *shape* —
  only whether the existing flags are enforced.
- Auto-removing the `replace` directive (that is the developer's choice
  when they move off the local checkout); item 2 only fixes the version the
  `require` line carries.
- Publishing `@dockyard/bridge` / `@dockyard/ui` to npm (separate
  V2-BACKLOG item; the bigger downstream gap) — out of scope here.
- A `--force` that overwrites colliding files. `--here` permits a non-empty
  dir but a file collision with a scaffold output stays a typed error.

## Acceptance criteria

### Item 1 — enforce the quality gates (D-168)

- [ ] `internal/validate` (and/or `internal/testgate`'s contract category)
      reads `manifest.RequireFixtures` and reports a Blocker for a declared
      tool missing its fixture set when the flag is `true`.
- [ ] `require_contract_tests: true` makes a project missing its contract
      test(s) a Blocker.
- [ ] Both flags `false` (or absent) → no new diagnostic (back-compat for a
      project that never opted in).
- [ ] The blank scaffold and both templates (which ship fixtures + tests)
      stay green under `validate` / `test`.
- [ ] A test deletes a fixture (and a contract test) from a scaffold with
      the flags on and asserts `validate` / `test` now fail with a clear,
      source-located diagnostic.

### Item 2 — scaffold `go.mod` version pin

- [ ] The scaffold writes a real published `require` version (a pinned
      `vX.Y.Z`, or the build's known version) instead of `v0.0.0`.
- [ ] With `--dockyard-path` set, the `replace` still points at the local
      checkout (unchanged build-from-source path); the `require` version is
      irrelevant to the build but valid.
- [ ] Without `--dockyard-path`, a generated project resolves the published
      module from the proxy and `dockyard validate` is green on the first
      try (the D-166 post-step still applies).
- [ ] Removing the `replace` from a scaffolded project leaves a `go.mod`
      whose `require` resolves — `go get …@<newer>` upgrades without a hand
      edit. Covered by the golden test update + an integration assertion.

### Item 3 — `dockyard new` target-dir UX

- [ ] `checkTarget` rejecting a non-empty dir names the entries it found
      (at least the first few) so the developer knows *what* blocked it.
- [ ] `dockyard new <name> --here` scaffolds into an existing non-empty dir;
      a scaffold output that would overwrite an existing file is still a
      typed error (no silent overwrite).
- [ ] `--here` appears in `dockyard new --help`; a smoke check asserts it.

- [ ] §19: the `scaffold-a-server` / `attach-a-ui-resource` / `validate`

## Risks / open questions

- **Enforcing item 1 can turn a green project red.** Mitigated: the
  default scaffold ships fixtures + tests; only a project that opted in and
  then removed them fails. Called out in CHANGELOG per D-168's semver note.
  **Open question:** does `require_fixtures` apply to *every* tool or only
  UI-bearing tools? Resolve against the manifest's existing fixture
  semantics during implementation and record it.
- **Item 2 version source.** Whether the scaffold pins a hardcoded version,
  reads it from a build-stamped `internal/version`, or writes `@latest`
  semantics — decide during implementation and file the decision (a
  hardcoded pin needs a release-time bump discipline; a build-stamp ties
  the scaffold to the CLI's own version, usually the right answer).
- **Version of this wave (1.2.1 vs 1.3.0).** Item 3's `--here` is additive
  (minor); item 1 is a bugfix that can change behaviour. Decide the release
  version at cut time per which items land (D-159).
