## Summary

Adds an opt-in `authz.Config.ExposeRawToken` (default `false`) plus a
`WithRawToken` / `RawTokenFromContext` accessor pair, so a resource server that
performs RFC 8693 token exchange can retrieve the **validated** inbound bearer
token from the handler context and present it as a `subject_token` to a trusted
authorization server. Additive and backwards-compatible: no existing signature
changes, and with the flag off the token is discarded after validation exactly
as today.

## RFC anchor

- RFC §19.2 — stateless OAuth resource-server support (amended in this wave to
  carve out the opt-in delegation exposure).
- RFC §15 — security posture (token handling).

## Briefs informing this phase

- brief 03 — Go MCP SDK audit (auth surface).
- brief 07 — MCP 2026-07-28 migration (authorization draft).

## Brief findings incorporated

- brief 07 — the protected MCP server is an OAuth 2.1 resource server that
  validates audience/resource binding on every request; the MCP prohibition is
  *token passthrough*, not RFC 8693 exchange to a trusted authorization server.
  Audience validation (`principal.Resource == cfg.Resource`) is already
  enforced and is the precondition for safe exposure.
- brief 03 — Dockyard builds on the go-sdk's resource-server surface; the raw
  bearer token is parsed once by `authz.ParseBearer` (plural-header / malformed
  handling). Exposing the *validated* token from the framework avoids a second,
  divergent parser in every consumer wrapper.

## Findings I'm departing from (if any)

Refines **D-195** ("never acquires, forwards, logs, or stores tokens") — its
blanket "never forwards" becomes "never performs token passthrough; discards by
default; opt-in validated-token exposure for RFC 8693 is permitted." Recorded in
**D-201** and the RFC §19.2 amendment. Not a silent departure.

## Goals

- A resource-server author can obtain the validated inbound token in a handler,
  gated behind an explicit opt-in, for RFC 8693 `subject_token` use only.
- The default posture (discard after validation) is unchanged for every existing
  server.
- The token never leaks into `obs/v1`, framework logs, or durable Task/MRTR
  continuation state.

## Non-goals

- Dockyard performing the RFC 8693 exchange itself, or any OAuth-client flow
  (Harbor's role; the handler is the RFC 8693 client).
- Exposing the token for any purpose other than an RFC 8693 `subject_token` to a
  trusted exchange (documented constraint, not a code-enforceable one).
- Carrying the token across a resumed/continued request (a resumed request
  re-exchanges on its own fresh inbound token).
- Changing any existing public signature.

## Public API surface

```go
// authz
type Config struct { /* … */ ExposeRawToken bool }
func WithRawToken(ctx context.Context, token string) context.Context
func RawTokenFromContext(ctx context.Context) (string, bool)
```

Additive only. `WithRawToken` is framework-internal (set by the server
middleware); `RawTokenFromContext` is the handler-facing read.

## Test plan

- **Unit (`runtime/authz`):** `WithRawToken`/`RawTokenFromContext` round-trip;
  empty token ⇒ `("", false)`; absent key ⇒ `("", false)`.
- **Integration (`runtime/server`, real validator/driver):** flag on ⇒ handler
  reads the exact bearer the client sent; flag off ⇒ `("", false)`; rejected
  request (bad token / wrong resource / insufficient scope) ⇒ handler never
  runs / no token; both lifecycles; a Task created on an ExposeRawToken request
  has no token bytes in its persisted store record or MRTR continuation blob.
- **Live verification:** `dockyard`-level or httptest end-to-end exercising the
  real HTTP middleware (not a unit shim) — see acceptance.

## Risks / open questions

- **Misuse surface.** Once exposed, the framework cannot force the handler to use
  the token only for RFC 8693. Mitigated by default-off, the doc warning, and the
  validated-only/never-logged/never-persisted invariants. This is the residual
  risk the opt-in accepts.
- **Durable-state leakage** is the load-bearing invariant — the token must never
  reach the Store or the MRTR continuation blob. Proven by an explicit
  token-free-state test, not by inspection.

## Glossary additions

- **Delegated token exchange** — obtaining a downstream token by presenting the
  validated inbound token as an RFC 8693 `subject_token` to a trusted
  authorization server; distinct from *token passthrough*. Add to
  `docs/glossary.md` in the implementing PR.
