## Summary

## Goals

- A Dockyard server author can advertise a logo, homepage, and description that a
  host MAY render, on both the legacy and stateless lifecycles.
- The default (unbranded) `serverInfo` is unchanged for every existing server.
- The manifest carries branding declaratively; the runtime keeps its no-manifest
  dependency (the two are kept in step by convention, not a runtime read).

## Non-goals

- Guaranteeing a host renders the icon â€” SEP-973 rendering is client-dependent.
- A manifestâ†’runtime auto-merge (would need a new public manifest-loader API or a
  codegen step); deferred to the V2 backlog (D-203).
- Per-tool / per-resource icons (a separate SEP-973 surface); out of scope here.

## Risks / open questions

- **Client rendering is out of Dockyard's control.** SEP-973 icons are a hint;
  the guide sets that expectation so a "logo not showing" report is triaged to
  the client, not misfiled as a Dockyard bug.
- **Manifest vs runtime source of truth.** Documented (D-203): manifest is
  declarative, runtime emits from `server.Info`, no hidden coupling. A public
  manifest-load bridge is a deferred, larger surface.
