## Summary

Two bounded inspector-side polish items that fell out of the V1 cut, picked
up in a single wave because both extend already-shipped seams without
widening them: (1) `dockyard dev` auto-attaches the inspector as a third
supervised child, with a `--no-inspector` opt-out (closes D-101); (2) the
inspector grows a Prompts rail tab + a `POST /api/prompts/get` operator-
invoke endpoint so the Phase 28 prompts surface is finally exercisable
through the local inspector (closes D-151). Both items keep P4 framing
intact (D-144 — operator-initiated only, dev-mode-gated, localhost-bound)
and stay within the existing `obs/v1` and `internal/inspector` contracts.

## RFC anchor

- RFC §12 — the inspector (the canonical scope; line 711 — "implements
  the host half of the `ui/` postMessage bridge to render Apps locally";
  the Prompts panel extends the same lone client-shaped surface).
- RFC §9.2 — the `dockyard dev` orchestrator (the embedded fsnotify dev
  loop; the auto-attach seam adds a third supervised child to the

## Brief findings incorporated

- From the original phase-19 work (D-085 / supervisor seam): the
  `internal/devloop` supervisor is an open seam for additional
  supervised children — adding the inspector as a third does not
  reshape it, only adds one supervisor instance.
- From the original phase-22/23 work (D-099 / D-103 / D-131 / D-144):
  the inspector's client-shaped surfaces are uniformly short-lived,
  per-request MCP client sessions opened by an operator UI action,
  with a documented decision entry. The Prompts panel extends the
  same pattern to `prompts/get`.
- From the original phase-28 work (D-151 / D-152): MCP prompt
  arguments are a flat string-keyed map (no JSON Schema layer), so
  the Prompts panel renders a simple typed form keyed by
  `PromptArgument.Name` — not the contract-first schema-form pipeline
  the Tools panel uses.

## Findings I'm departing from (if any)

None.

## Goals

1. A developer running `dockyard dev` gets the inspector started
   automatically alongside the Go server + Vite, with the inspector URL
   printed to stdout so the developer can cmd-click to open it.
2. The `--no-inspector` flag lets a CI / headless dev run opt out
   without losing the rest of the dev loop.
3. The inspector renders a Prompts panel that lists the attached
   server's prompts and lets an operator pick + invoke one against the
   attached server, rendering the returned messages.
4. Every existing P4 invariant (dev-mode-gated, localhost-bound,
   operator-initiated only — D-144) continues to hold; the Phase 27
   `internal/inspector` `mcp.NewClient` security audit allow-list
   gains exactly one entry (the new prompts client).

## Non-goals

- Reshaping `internal/devloop`'s supervisor model.
- Extending contract-first to prompts (D-152: prompt arguments are a
  flat string map; contract-first does not apply).
- Adding a long-lived MCP client anywhere — every prompts/get call
  opens a fresh, short-lived session and closes it (the D-103 / D-131
  / D-134 pattern).
- Rendering richer prompt-message content variants (resource embeds,
  images). The runtime API exposes the text-only PromptMessage shape
  for V1 (D-151); the panel renders that shape verbatim. A future
  richer variant is additive.
- Auto-restarting the inspector on a Go-server restart. The inspector
  re-attaches lazily through its existing relay reconnect path — same
  shape as the Vite child surviving a server restart today.

## Acceptance criteria

      stdout once the supervisor reports it ready.
- [ ] `dockyard dev --no-inspector` skips the inspector child cleanly;
      the Go server + Vite remain supervised; no inspector port is

## Public API surface

- `devloop.Config.DisableInspector bool` — opt-out flag, false by default
  (auto-attach by default; the V2-backlog "definition of done" makes the
  default-on choice).
- `devloop.Config.InspectorAddr string` — optional override of the
  inspector's bind address; empty selects `127.0.0.1:0` (OS-assigned).
- `inspector.PromptSource func(ctx) ([]PromptInfo, error)` — read-only
  source for the prompts list; mirrors `AppSource`.
- `inspector.PromptInvoker func(ctx, PromptGetRequest) (*PromptGetResponse, error)`
  — operator-initiated `prompts/get`; mirrors `ToolInvoker`.
- `inspector.PromptsFromServer(baseURL string) (PromptSource, PromptInvoker)`
  — constructs both from a running MCP server's base URL; mirrors
  `ToolsFromServer`.

## Risks / open questions

- **Risk: Vite + inspector port collision on a developer with a custom
  Vite port already on 5173.** The inspector defaults to port 0
  (OS-assigned), so the collision is structurally impossible — the
  OS hands the inspector a free port. A developer who pins
  `--inspector-port` to a busy port gets the same typed bind
  rejection any other `dockyard inspect` would.
- **Risk: a Go-server restart races the inspector's relay.** The
  relay already reconnects lazily through its existing SSE-reconnect
  path; the inspector is reused across restarts.
- **Risk: prompts/list pagination.** The SDK supports cursor
  pagination; the V1 panel walks the first page only. A prompts-heavy
  server would surface a follow-up V2 item (small, additive).
