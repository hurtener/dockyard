## Summary

Two items deferred from v1.3 wave A, now their own focused wave:

1. **Item 5 ‚Äî bridge View-side task-progress channel.** `runtime/tasks.
   TaskHandle.Progress` exists server-side, but the Svelte bridge
   (`web/bridge`) exposes no View-side progress channel, so an App's card
   cannot render a live "62%". Add a `ui/notifications/task-progress` host‚Üí
   View notification + a typed `onTaskProgress` subscribe helper (mirroring
   `onToolResult` / `onHostContextChanged`); emit the runtime's mid-task
   progress as an `obs/v1` `task.progress` `PhaseProgress` event so the
   inspector host-bridge can forward it to the View; degrade cleanly when a
   host does not forward progress.

2. **Item 4 ‚Äî publish `@dockyard/bridge` + `@dockyard/ui` to npm.** Today
   the two packages are workspace-only (`main: ./src/index.ts`, version
   `0.1.0`, no build, no `publishConfig`), so every UI scaffold needs the
   hidden `--dockyard-path` flag + a local checkout. Make each consumable
   from npm (build/exports), bump them to track the repo version, verify
   with `npm pack` + a scaffold install, add a gated tag-push publish job,
   and resolve the scaffold's `__DOCKYARD_*_SPEC__` tokens to the published
   versions when `--dockyard-path` is omitted.

This plan covers both; **PR A ships item 5, PR B ships item 4** (see the
PR-plan note above).

## RFC anchor

- RFC ¬ß7.3 ‚Äî the Svelte bridge shell library (the View side of the `ui/`
  postMessage dialect). Item 5 adds a host‚ÜíView notification + a View-side
  subscribe helper to it.
- RFC ¬ß8 ‚Äî MCP Tasks (`TaskHandle.Progress`). Item 5 surfaces a task's
  progress fraction into the App's iframe.
- RFC ¬ß11.2 / ¬ß11.3 ‚Äî `obs/v1`. Item 5 emits a mid-task `task.progress`
  `PhaseProgress` event (the reserved-but-unused phase) carrying the
  fraction; an additive, documented field change to `TaskProgressPayload`.
- RFC ¬ß12 ‚Äî the inspector. Item 5 wires the inspector host-bridge to
  forward task progress to the App preview so it is demoable through
  `dockyard inspect`.
- RFC ¬ß14 ‚Äî packaging & release. Item 4 publishes the two frontend
  packages alongside the release pipeline.
- RFC ¬ß7.5 ‚Äî capability-driven degradation (never a host matrix). Item 5's
  degradation story.

## Brief findings incorporated

- **brief 01 ¬ß2.4 ‚Äî the host‚ÜíView notification set is the bridge's seam.**
  Item 5 adds `task-progress` to that set the same way `tool-result` /
  `host-context-changed` are modelled (a typed `Topic`, a router case, an
  `onTaskProgress` helper) ‚Äî not a bespoke channel.
- **brief 01 ¬ß2.4 / RFC ¬ß7.5 ‚Äî capability-driven degradation, never a host
  matrix.** Item 5 degrades by absence: a host that never forwards
  `task-progress` simply never fires `onTaskProgress`; no per-host matrix,
  no negotiated capability flag required (mirrors the elicitation channel,
  D-134).
- **brief 05 ¬ß3.1 ‚Äî `PhaseProgress` is the reserved intermediate-point
  phase.** `obs/event.go` already declares `PhaseProgress` "reserved for
  future mid-task updates"; item 5 makes `TaskHandle.Progress` emit it, the
  intended use.
- **brief 04 ‚Äî first-run friction is the headline DX risk.** Item 4
  removes the Go/npm asymmetry: the Go module installs from the proxy while
  the frontend half silently needs a checkout. After item 4 a `--template`
  scaffold's `web/` `npm install` succeeds with no `--dockyard-path`.

## Findings I'm departing from (if any)

None. Both items implement documented intent (the wave-A plan's items 4/5
spec + the two V2-BACKLOG entries) and the reserved `obs/v1` `PhaseProgress`
extension point.

## Goals

1. **Item 5.** An MCP App can subscribe to live task progress
   (`bridge.onTaskProgress((p) => ‚Ä¶)`), typed, and render a "62%" card;
   the inspector forwards a running task's progress to the App preview so
   it is demoable through `dockyard inspect`; a host that does not forward
   progress degrades cleanly (the subscriber never fires).
2. **Item 4.** `@dockyard/bridge` and `@dockyard/ui` are npm-consumable by
   a downstream Svelte/Vite build; their version tracks the repo/Go-module
   version; a tag-push CI job publishes both (`--access public`,
   idempotent) gated on an `npm pack` + scaffold-install verification; a
   `--template` scaffold's `web/` `npm install` succeeds with **no**
   `--dockyard-path` and no checkout.

## Non-goals

- **Item 5.** No change to the polled `tasks/get` `StatusMessage` shape (a
  poller still sees the "62% ‚Äî ‚Ä¶" formatted message); no new `web/ui`
  progress-bar component required for the demo (the inspector's TasksPanel
  already renders the lifecycle via `Timeline`; an App renders its own card
  from the forwarded value). No View‚Üíhost progress (progress is host‚ÜíView
  only ‚Äî P4: the App is a View).
- **Item 4.** No npm publish executed by this work ‚Äî the CI job is added
  and stays inert until a maintainer tags a release. No change to the
  `--dockyard-path` build-from-source path (it stays a convenience; the
  `replace` still wins when set). No npm workspaces / root `package.json`
  reshape (each package stays independently installable, as today).

## Acceptance criteria

### Item 5 ‚Äî bridge View-side task-progress channel

- [ ] **Protocol.** `web/bridge/src/protocol.ts` gains
      `HostNotification.taskProgress = 'ui/notifications/task-progress'`
      and a typed `TaskProgressParams` (taskId, optional fraction 0‚Äì1,
      optional message, optional status). Exported from the package barrel.
- [ ] **View subscribe helper.** `BridgeShell.onTaskProgress((p:
      TaskProgressParams) => void): Unsubscribe`, fed by a
      `NotificationRouter` `Topic`/dispatch case + an `onTaskProgress`
      router method ‚Äî mirroring `onToolResult` / `onHostContextChanged`.
- [ ] **Runtime emission.** `TaskHandle.Progress` emits an `obs/v1`
      `task.progress` event at `PhaseProgress` carrying the clamped
      fraction (a new optional `Fraction *float64` on `TaskProgressPayload`)
      and the raw message; emitted only when the status update succeeds
      (a `Progress` call on a non-working task still errors and emits
      nothing). `TaskHandle.Status` emits a `PhaseProgress` event with the
      message and no fraction.
- [ ] **obs/v1 contract.** The `Fraction` addition is additive (existing
      consumers ignore it), documented in `CHANGELOG.md`, and the obs
      golden tests are re-pinned to the new shape; `SchemaVersion` stays
      `dockyard.obs/v1` (an additive field, not a shape break).
- [ ] **Inspector forwarding.** The inspector host-bridge gains
      `sendTaskProgress(params)`; the inspector wires the `obs/v1`
      `task.progress` stream (the `PhaseProgress` events) to the App
      preview's host-bridge so a running task's progress reaches the View.
- [ ] **Degradation.** A host that never sends `task-progress` ‚Üí
      `onTaskProgress` never fires; no error, no host matrix (RFC ¬ß7.5).
- [ ] **Tests.** The bridge vitest suite covers: dispatch routes
      `task-progress` to `onTaskProgress`; the payload is typed; unsubscribe
      works; an unknown/absent notification is a no-op. Runtime `-race`
      tests assert `Progress` emits a `PhaseProgress` event with the
      fraction and `Status` emits one without; a `Progress` on a terminal
      task emits nothing.

### Item 4 ‚Äî publish `@dockyard/bridge` + `@dockyard/ui` to npm

- [ ] **Packaging.** Each package is consumable from npm by a downstream
      Svelte/Vite build ‚Äî `@dockyard/ui` via `@sveltejs/package` (a build
      emitting `dist/` + a `svelte` export condition), `@dockyard/bridge`
      via a `tsup`/`tsc` build (emitting `dist/` JS + `.d.ts`);

- [ ] **Version.** Both bump off `0.1.0` to track the repo version

- [ ] **Verify before publish.** `npm pack` each + an install-into-a-fresh

## Risks / open questions

- **obs/v1 emission volume.** A chatty handler calling `Progress` in a
  tight loop emits one `obs/v1` event per call. The emit path is
  non-blocking (ring buffer / bounded fan-out ‚Äî P2), so it never blocks the
  handler, but a pathological caller could flood the stream. Mitigation:
  document `Progress` as advisory + best-effort (it already is); no
  rate-limit in V1 (the inspector folds by task id). Recorded in D-171.
- **obs/v1 shape change discipline.** Adding `Fraction` to
  `TaskProgressPayload` touches the obs/v1 contract (¬ß8). It is additive
  (optional, omitempty) ‚Äî existing consumers are unaffected ‚Äî but it MUST
  be documented (CHANGELOG + D-171) and the golden tests re-pinned, never a
  silent edit.
- **Item 4 first publish is semi-irreversible.** Mitigated by the
  `npm pack` + scaffold-install verification gating the job, the
  `--access public` + skip-already-published idempotency, and a
  maintainer-driven first tag (this work never publishes).
- **`@sveltejs/package` vs raw-source-with-`svelte`-condition for `ui`.**
  A built `dist/` is the safer, spec-blessed path for a publishable Svelte
  component library (a downstream consumer's bundler resolves the `svelte`
  condition to pre-processed components). Decide the exact build at
  implementation time and record it in D-172; the acceptance gate
  (`npm pack` + scaffold build) is the real test either way.
