## Summary

Adds an opt-in `authz.Config.UnauthenticatedHandshake` (default `false`) that
serves the MCP lifecycle and discovery methods **without** a token and requires a
valid token only on invocations. Additive and backwards-compatible: no existing
signature changes, and with the flag off every method is protected exactly as
today (the middleware does not even read the body). Unblocks a multi-user MCP
runtime that opens one shared connection and runs `initialize` / `server/discover`
/ `tools/list` with no per-user token — the token is per-user and exists only at
tool-call time.

## RFC anchor

- RFC §19.2 — stateless OAuth resource-server support (amended in this wave to
  carve out the opt-in unauthenticated handshake).
- RFC §15 — security posture (authorization boundary).

## Briefs informing this phase

- brief 03 — Go MCP SDK audit (HTTP transport / auth surface).
- brief 07 — MCP 2026-07-28 migration (stateless lifecycle, `server/discover`).

## Brief findings incorporated

- brief 07 — the modern 2026-07-28 stateless lifecycle uses `server/discover` as
  a single combined handshake+discovery method; the legacy 2025-11-25 lifecycle
  uses `initialize` + `*/list`. Both are exempt; invocations (`tools/call`,
  `resources/read`, `prompts/get`, `tasks/*`) are unchanged and stay protected.
- brief 03 — Dockyard sets HTTP security explicitly and the MCP POST body is
  already bounded and buffered by `mcpRequestBodyLimit` (4 MiB) *outside* the
  authorization middleware, so the method peek reads an in-memory buffer and adds
  no unbounded read.

## Findings I'm departing from (if any)

None. This extends the discovery-public trade of **D-152** (contract-first stops
at discovery; prompts/resources registration is public) from stdio ergonomics to
the HTTP authorization boundary. Recorded in **D-202** and the RFC §19.2
amendment. Not a silent departure.

## Goals

- A resource server can accept the MCP handshake and discovery on one shared,
  unauthenticated connection, while every invocation still requires a valid,
  audience-bound token.
- The default posture (every method protected) is unchanged for every existing
  server.
- The exempt set is Dockyard-owned and deny-by-default: no invocation method can
  ever be exposed, and unknown/future methods fail closed to token-required.

## Non-goals

- Making any existing behavior less strict on invocations — signature, issuer,
  audience/resource, subject, and required scopes are enforced unchanged.
- A caller-configurable exempt set (it is a package constant, by design).
- Binding the SSE stream to a per-session principal, or exempting benign control
  notifications (`logging/setLevel`, `notifications/cancelled`,
  `notifications/roots/list_changed`) — deliberately out of scope (D-202).
- A `skills/*` classification — there is no such MCP route (Dockyard's `skills/`
  are Agent Skills, not a JSON-RPC method).

## Risks / open questions

- **Discovery becomes public.** Exempting `tools/list` makes tool names/schemas/
  descriptions readable without a token. This is the intended trade (D-152), it
  is opt-in, and it is documented on the field and in the guide. A server that
  must keep discovery private leaves the flag off.
- **Fail-open would be a security bug.** The load-bearing invariant is that the
  peek never classifies an invocation as exempt. Proven by the deny-by-default
  allowlist (unknown ⇒ protected), the batch-any-invocation rule, and the
  `FuzzAllMethodsExempt` fail-closed target — not by inspection.
