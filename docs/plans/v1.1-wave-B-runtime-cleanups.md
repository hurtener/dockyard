## Summary

Wave B closes two recorded post-V1 deferrals from `docs/V2-BACKLOG.md`,
both runtime-side and both bounded:

1. **D-108 â€” scaffold + `dockyard run` auto-wire of `tasks.Engine`.** A
   project whose manifest declares any tool with
   `task_support: optional` or `required` now gets a working
   `tasks.NewInMemoryStore()` + `tasks.NewEngine(...)` constructed in
   the scaffolded `main.go` and attached via `server.Options{Tasks:
   engine}`. The blank scaffold (whose example tool declares
   `task_support: forbidden`) and the `analytics-widgets` template
   (whose tools all declare `forbidden`) keep their engine-free
   `main.go`; the `approval-flows` template (whose hand-written
   wiring already follows this shape) goes through the same
   generator path so the wiring is one piece of truth. `dockyard
   run` reads the project's manifest at start time and surfaces a
   warning when the manifest declares task-supporting tools but the
   project's `main.go` source does not appear to wire an engine â€”
   the "detection runs at scaffold time and at run time, and they
   must agree" half of the V2-backlog DoD.
2. **D-145 â€” Claude signed-origin testgate cleanup.** The capability-
   degradation category in `internal/testgate/categories.go`
   currently passes a hardcoded synthetic URL
   (`https://capability-test.example/mcp`) to `DeriveDomain` so the
   Claude host profile's signed-origin invariant does not trip. The
   workaround retires: the host-profile interface gains a
   `RequiresServerURL() bool` method (Path B in the V2-backlog), so
   the capability category short-circuits a profile that requires a
   server URL rather than testing it with an empty or synthetic one.
   The capability test still exercises the seam end-to-end for every
   profile that can derive without a server URL (generic) and proves
   each signing profile (claude) declares the requirement honestly.

## Goals

- `dockyard new` of a project whose manifest declares any tool with
  `task_support: optional` or `required` produces a `main.go` that
  constructs `tasks.NewInMemoryStore()` + `tasks.NewEngine(...)` and
  attaches it via `server.Options{Tasks: engine}` â€” no hand edit
  required, the lifecycle defaults are sensible, the engine starts
  and stops cleanly on Ctrl-C.
- The blank scaffold (whose `greet` tool declares
  `task_support: forbidden`) and the `analytics-widgets` template
  (whose three widget tools all declare `forbidden`) keep their
  engine-free `main.go` â€” zero overhead, no regression.
- The `approval-flows` template's `main.go` continues to wire the
  engine (it always has). Its hand-written wiring is brought into
  the shape the generator emits, so a future maintainer reading
  the two does not have to puzzle out two engine-construction
  idioms.
- `dockyard run` reads the project's manifest at start time and, if
  the manifest declares task-supporting tools but the project's
  `main.go` source does not appear to attach `server.Options.Tasks`,
  surfaces a clear warning on stderr â€” the run still proceeds (the
  project's main is authoritative).
- The capability-degradation testgate category exercises every
  registered host profile honestly: a profile that declares
  `RequiresServerURL() == true` is exempt from the empty-URL
  derivation (the seam is proven by the profile's own tests); a
  profile that declares `RequiresServerURL() == false` derives
  cleanly against an empty URL. The `syntheticServerURL` workaround
  is removed from `internal/testgate/categories.go`.
- The capability test continues to pass against the `analytics-
  widgets` scaffold and against every other shipped project.
