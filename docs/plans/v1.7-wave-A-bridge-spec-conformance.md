## Summary

The v1.6.1 post-mortem (D-179/D-180/D-181) fixed three handshake bugs that made
a wire-correct Dockyard App render blank in Claude Desktop. All three shared one
root cause: **`web/bridge` hand-transcribes the MCP Apps `ui/` wire dialect**
(method names, notification shapes, capability and host-context types, the
handshake lifecycle) instead of deriving it from the official spec. A
hand-transcription drifts the moment the spec moves or a field is mis-copied,
and nothing fails until a strict host rejects it in production.

A follow-up diff of `web/bridge` against the official
`@modelcontextprotocol/ext-apps` schema (`src/generated/schema.ts`) found the
three fixed bugs were not the only drift ‚Äî there are **four more shape/behaviour
divergences and one by-design extension boundary** still latent.

This wave does the structural fix and the cleanup together:

1. **Adopt the official ext-apps schema as the View wire layer's single source
   of truth (option B). (D-182 ‚Äî amends RFC ¬ß7.3, extends ¬ß10/¬ß16.)** Vendor the
   machine-readable `@modelcontextprotocol/ext-apps` schema by upstream SHA
   (alongside the existing `mcp-apps-2026-01-26.mdx`), derive the bridge's
   protocol types from it by type-inference, and add a conformance test layer
   that parses the bridge's outbound wire ‚Äî and the inspector host's inbound
   handling ‚Äî against the vendored schema. The shipped App bundle stays
   **Zod-free**: the schema is `import type` for inference and `.parse()` only in
   tests. This is the **¬ß8.2 Tasks pattern** (wire layer code-generated/derived
   from a vendored schema snapshot, isolated behind one seam) applied to the
   View half.

2. **Conformance fixes the schema forces into the open.** Four divergences the
   diff found, each now caught by the conformance layer:
   - **A ‚Äî `appCapabilities.availableDisplayModes`** (we emit `displayModes`; Zod
     `.object()` strips the unknown key, so a host silently never learns the App
     supports fullscreen/pip).
   - **B ‚Äî `ui/resource-teardown` is a Request, not a notification** (we `close()`
     without responding; a spec host waits for the response). Adds the
     app-initiated `ui/notifications/request-teardown`.
   - **C ‚Äî `containerDimensions` flexible sizing** (`{width|maxWidth,
     height|maxHeight}`; we model fixed `{width, height}` only).
   - **D ‚Äî host fonts** (`styles.css.fonts` + the `applyHostFonts` behaviour; we
     read `styles.variables` only).

3. **Fence the Dockyard Tasks√óApps View notifications as explicit extensions.
   (D-183.)** `ui/notifications/task-progress` and
   `ui/notifications/elicitation-response` are **not** in the MCP Apps schema ‚Äî
   they are Dockyard Tasks√óApps extensions (RFC ¬ß8, D-134). They function only
   against a Dockyard-aware host (the inspector, Harbor); a stock host ignores
   them. Move them to a clearly-separated `dockyard-ext` surface outside the
   conformed namespace, assert (in conformance) that they are the *only* non-spec
   messages, and document the non-portability.

4. **Harden the inspector host against the same schema (the D-181 follow-up).**
   The inspector accepted every non-spec View shape (it read our wrong
   `displayModes`, sent `initialized` host‚ÜíView, treated teardown as fire-and-
   forget) ‚Äî which is exactly why all three v1.6.1 bugs passed locally. The
   inspector host now validates inbound View messages against the vendored
   schema and behaves as a faithful spec host: it resolves ready on **receiving**
   the View's `initialized` (it no longer sends one), and sends
   `ui/resource-teardown` as a request that awaits the View's response.

### Why these, and the triage of the rest

The diff surfaced six divergence classes. Triage (mirroring the v1.6 wave's
`[CONFIRMED]/[ALIGN]/[OPEN]` lens):

- **Items 1, A‚ÄìD, the inspector** ‚Äî confirmed shape/behaviour drift vs the
  schema, all spec-grounded. In scope.
- **The Tasks√óApps notifications** ‚Äî not drift; Dockyard extensions by design.
  Fenced + documented (item 3 / D-183), not "fixed."
- **Missing additive features ‚Äî View-side `resources/read`, `ping`,
  `notifications/message` (logging)** ‚Äî the schema defines these; we don't
  implement them. They are **additive capability gaps, not conformance bugs**
  (a host does not require an App to support them). **Deferred** to a backlog
  entry, not gated into this wave ‚Äî see Non-goals. Including them would expand a
  conformance wave into a feature wave.

## RFC anchor

- **RFC ¬ß7.3 ‚Äî the bridge shell library.** This wave **amends ¬ß7.3**: the bullet
  list of what the shell does is unchanged in spirit, but the section gains the
  binding statement that the shell's wire layer (methods, notification shapes,
  capability + host-context types, handshake lifecycle) is **derived from and
  conformance-tested against the vendored official ext-apps schema**, not
  hand-maintained ‚Äî the View-side analogue of the ¬ß8.2 Tasks rule and the ¬ß5.4
  `protocolcodec` seam.
- **RFC ¬ß8.2 ‚Äî the Tasks shim.** The precedent this wave generalises: "the wire
  layer **code-generated from a vendored schema snapshot** and isolated in
  `internal/protocolcodec`." Item 1 applies the identical discipline to the View
  bridge (the seam is `web/bridge`'s `protocol.ts`, the schema is vendored
  ext-apps).
- **RFC ¬ß10 / ¬ß16 ‚Äî forward-compatibility by isolation.** ¬ß10 already mandates
  vendoring **every MCP spec by upstream SHA + date** into the repo. This wave
  extends that to the **machine-readable** ext-apps schema (the `.ts`, alongside
  the already-vendored `.mdx`), and makes the View wire layer a consumer of it ‚Äî
  so a spec bump is "re-vendor the SHA, re-run the conformance/golden diff," not
  "re-read the prose and hope."
- **RFC ¬ß7.5 ‚Äî capability-driven degradation.** Item A (`availableDisplayModes`)
  restores the capability the host reads to drive display-mode degradation; the
  fix makes ¬ß7.5's mechanism actually work against a strict host. No ¬ß7.5 text
  change.

## Briefs informing this wave

- **brief 01 (MCP Apps spec teardown)** ‚Äî ¬ß2.4 the `ui/` postMessage dialect,
  the handshake lifecycle, `hostContext` shape; the authority for what the
  bridge must conform to. The vendored schema this wave adds is the
  machine-readable companion to the brief's prose reading.

## Brief findings incorporated

- **brief 01 ¬ß2.4 ‚Äî the View is the handshake initiator and the dialect is one
  contract shared by the View and the host half.** The post-mortem proved the
  brief's reading was right and the *implementation* drifted from it; item 1
  makes the schema, not a hand-reading, the enforced contract ‚Äî so the brief's
  intent becomes mechanically checkable.

## Findings I'm departing from (if any)

- **Departure from the implicit "hand-derive the View wire layer" stance.** No
  single decision said "hand-write `protocol.ts`"; it was the path of least
  resistance when the bridge phase (Phase 11) predated a stable published
  ext-apps schema. RFC ¬ß8.2 had *already* chosen "vendor + generate" for the
  Tasks wire layer; the View bridge simply never got the same treatment. D-182
  makes the ¬ß8.2 discipline binding for the View half and supersedes the implicit
  hand-derive stance. History is not rewritten ‚Äî Phase 11 stands as the record of
  what was built then; D-182 is the correction.
- **No departure from the lean-bundle rule (RFC ¬ß7.4).** Option B is sometimes
  read as "just `import @modelcontextprotocol/ext-apps`," which would ship Zod
  into every single-file App bundle. This wave explicitly does **not** do that:
  the vendored schema is type-only at runtime (`import type` ‚Üí erased) and
  `.parse()`'d **only in tests**; a bundle guard asserts the App output carries
  no Zod. The runtime stays as lean as today.

## Goals

1. The official ext-apps machine-readable schema is **vendored into the repo,
   pinned by upstream commit SHA + date**, and is the declared source of truth
   for the View wire layer (RFC ¬ß10 vendoring, applied to the `.ts`).
2. `web/bridge`'s protocol types (`InitializeParams`, `HostContext`, the
   capability shapes, every method/notification param) are **derived by
   type-inference from the vendored schema** ‚Äî a mis-named or wrong-shaped field
   is a `tsc` failure, not a runtime drift.
3. A **conformance test layer** parses the bridge's outbound wire (the
   `ui/initialize` params and every View‚Üíhost request/notification) against the
   vendored schema; a test asserting the `ui/initialize` params parse clean is
   the regression that would have caught D-179 (and catches item A).
4. The shipped App bundle is **Zod-free** ‚Äî a bundle-size/content guard asserts
   no schema-validation runtime leaked into the single-file output.
5. The four conformance fixes (A `availableDisplayModes`, B teardown-as-request +
   `request-teardown`, C flexible `containerDimensions`, D host fonts) land, each
   with a test, each green against the vendored schema.
6. The two Dockyard Tasks√óApps notifications are fenced into a separate
   `dockyard-ext` surface, documented as non-portable, and conformance asserts
   they are the only non-spec messages the bridge emits.
7. The **inspector host** validates inbound View messages against the vendored
   schema and behaves as a faithful spec host (ready-on-receive-`initialized`,
   no host-sent `initialized`, teardown-as-request). The host-bridge test that
   asserts an outbound host‚ÜíView `initialized` is replaced.
8. RFC ¬ß7.3 amended; `docs/decisions.md` (D-182, D-183); `docs/glossary.md`;
   CHANGELOG `[Unreleased]`; the `attach-a-ui-resource` skill + docs-site Apps
   guide (the Tasks-extension non-portability note, item 5); ¬ß19 surfaces in sync
   in the same PR.

## Non-goals

- **Shipping the official SDK runtime in the App bundle.** Vendoring the schema
  for types + test conformance only; never `.parse()` on a runtime App path
  (RFC ¬ß7.4 lean single-file bundle). The runtime bridge stays hand-written,
  Zod-free, and Svelte-idiomatic ‚Äî that binding layer is Dockyard's legitimate
  value-add over the framework-neutral SDK.
- **View-side `resources/read`, `ping`, `notifications/message` (logging).**
  Additive spec capabilities the bridge does not implement. Not conformance bugs
  (a host does not require them of an App). Recorded as a new `docs/V2-BACKLOG.md`
  entry ("bridge additive-capability parity"); revisited when an App actually
  needs them.
- **Replacing the public `dockyard-bridge` API.** `createBridge`, the Svelte
  stores, `callContract`, the `on*` subscriptions stay source-compatible ‚Äî this
  is an internals rewire behind a preserved facade, not a consumer-facing API
  break. (If any public type *shape* changes because it is now schema-derived ‚Äî
  e.g. `containerDimensions` gaining `maxWidth/maxHeight` ‚Äî that is additive and
  called out in CHANGELOG.)
- **A code generator for the TS types.** Type-*inference* from the vendored Zod
  schema (`z.input`/`z.infer`) is preferred over a bespoke `.ts` generator ‚Äî
  zero generator to maintain, and the types are the spec by construction. (Lock
  in D-182; the alternative is recorded as an open question.)

## Acceptance criteria

### Item 1 ‚Äî vendor the schema + derive types + conformance layer (D-182)

- [ ] The official `@modelcontextprotocol/ext-apps` machine-readable schema
      (`src/generated/schema.ts` and any transitive base-schema imports) is
      vendored into the repo at a single home, with a header recording the
      **upstream commit SHA + date** (matching the `docs/specifications/*` pin
      convention, RFC ¬ß10). The vendored file is not edited by hand.
- [ ] `web/bridge/src/protocol.ts` derives its wire types from the vendored
      schema by inference (`z.input`/`z.infer`) for at least: `InitializeParams`,
      `InitializeResult`, `HostContext`, `AppCapabilities`, `HostCapabilities`,
      and every method/notification param the bridge sends or reads. A
      deliberately wrong field name fails `tsc` (proven by a type-level test or a
      commented red example).
- [ ] A conformance test (`web/bridge`) `.parse()`s the bridge's emitted
      `ui/initialize` params and every View‚Üíhost request/notification payload
      against the vendored schemas and passes. The `ui/initialize` case is the
      explicit D-179 regression.
- [ ] **The shipped App bundle is Zod-free.** A build/bundle guard (a size or
      content assertion on a representative App's single-file output, e.g. the
      `analytics-widgets` build) confirms no Zod / schema-validation runtime is
      present. The vendored schema is reachable only via `import type` and test
      files.
- [ ] No change to the public `createBridge` / store / `on*` / `callContract`
      surface (source-compatible). Any additive type-shape change is listed in
      `Public API surface`.

### Item A ‚Äî `appCapabilities.availableDisplayModes`

- [ ] The bridge emits `appCapabilities.availableDisplayModes` (not
      `displayModes`) in `ui/initialize`; conformance `.parse()` confirms the key
      is the schema's. The inspector host reads `availableDisplayModes`.
- [ ] Display-mode negotiation tests (existing `bridge.test.ts`) updated to the
      new key; a host that advertises `fullscreen`/`pip` now sees the App's
      supported modes (the silent-strip is gone).

### Item B ‚Äî `ui/resource-teardown` is a request; add `request-teardown`

- [ ] The bridge handles host‚ÜíView `ui/resource-teardown` as a **request**:
      it runs cleanup, **responds** with the schema's result shape
      (`McpUiResourceTeardownResult`), then tears down ‚Äî it no longer treats it
      as a fire-and-forget notification.
- [ ] The bridge exposes the app-initiated `ui/notifications/request-teardown`
      (View‚Üíhost) helper for an App that wants to ask the host to tear it down.
- [ ] Tests: a mock host sends `ui/resource-teardown` (request) and receives the
      View's response before teardown; the View-initiated `request-teardown`
      notification is asserted on the wire.

### Item C ‚Äî flexible `containerDimensions`

- [ ] `HostContext.containerDimensions` is the schema shape
      (`{width?, maxWidth?, height?, maxHeight?}`); the host-context store and any
      consumer handle a host that sends `maxHeight/maxWidth` (flexible) as well as
      fixed `width/height`. A host-context-changed patch with `maxHeight` is
      reflected without dropping to `undefined`.

### Item D ‚Äî host fonts

- [ ] `HostContext.styles` carries `css.fonts` (the schema shape); the bridge
      **applies** host-provided font CSS (the `applyHostFonts` behaviour ‚Äî inject
      the `@font-face`/`@import` CSS into the View document) so host fonts load.
      A test asserts the font CSS is applied when present and is a no-op when
      absent.

### Item 3 ‚Äî fence the Dockyard Tasks√óApps notifications (D-183)

- [ ] `ui/notifications/task-progress` and `ui/notifications/elicitation-response`
      live in a clearly-named `dockyard-ext` module (not the conformed
      `protocol.ts` surface), documented as **Dockyard extensions outside the MCP
      Apps schema** that function only against a Dockyard-aware host.
- [ ] Conformance asserts these two are the **only** View‚Üíhost messages the
      bridge emits that are *not* in the vendored schema ‚Äî no accidental
      additional non-spec surface.
- [ ] The `attach-a-ui-resource` skill + docs-site Apps guide note that
      Tasks-augmented App behaviour (progress, elicitation) is Dockyard-host-only
      and a stock host (e.g. Claude Desktop) ignores it.

### Item 4 ‚Äî inspector host spec-conformance (D-181 follow-up)

- [ ] The inspector host validates inbound View messages (`ui/initialize` params,
      View‚Üíhost notifications) against the vendored schema; a non-spec shape is
      rejected (in tests).
- [ ] The inspector host resolves `ready` on **receiving** the View's
      `ui/notifications/initialized`; it **no longer sends** a host‚ÜíView
      `initialized`. The host-bridge test asserting an outbound `initialized`
      (`host-bridge.test.ts:96`) is replaced with one asserting ready-on-receive.
- [ ] The inspector host sends `ui/resource-teardown` as a **request** and awaits
      the View's response before tearing down the preview iframe; it handles the
      View‚Üíhost `ui/notifications/size-changed` to size the preview.

  surface.

## Risks / open questions

- **`schema.ts`'s transitive imports.** The upstream `src/generated/schema.ts`
  may import a base MCP schema (e.g. shared JSON-RPC / `Implementation` types).
  Vendoring must be self-contained (vendor the closure, or inline the few base
  types). Resolve at implementation; record the exact vendored set in D-182.
- **Infer vs generate.** Recommendation: **type-inference** (`z.input`/`z.infer`
  from the vendored Zod schema) over a bespoke `.ts` code generator ‚Äî no
  generator to own, types are the spec by construction. The risk is Zod-version
  coupling in the type graph (build-time only). Lock in D-182.
- **Keeping the runtime Zod-free.** The single real constraint: the App bundle
  must not ship Zod. Enforced by `import type`-only in runtime paths + `.parse()`
  in tests only + the bundle guard (acceptance item 1.4). If a build tool fails
  to tree-shake a type-only import, fall back to a thin generated `.d.ts` instead
  of importing the schema module. Note in D-182.
- **devDep: vendored file vs npm `@modelcontextprotocol/ext-apps`.**
  Recommendation: **vendor the file** (RFC ¬ß10 culture ‚Äî pin by SHA, hermetic, no
  registry at build time), with the npm devDep as the fallback if the schema's
  import closure is impractical to vendor. Lock in D-182.
- **Migration churn (inspector + bridge tests).** The public bridge API is
  preserved as a facade, but `protocol.ts`, the inspector host, and several test
  files are rewired. Sized as one wave; the v1.6.1 work already touched the same
  files, so the blast radius is known.
- **`request-teardown` vs `resource-teardown` naming.** The schema has both a
  host‚ÜíView `ui/resource-teardown` **request** and a View‚Üíhost
  `ui/notifications/request-teardown` **notification** (`schema.ts:437` vs `:489`)
  ‚Äî easy to conflate. The conformance layer keys on the exact wire strings to
  keep them distinct.

## Glossary additions / changes

- **Bridge shell library** (rewrite) ‚Äî note that the wire layer (methods,
  notification shapes, capability + host-context types, handshake lifecycle) is
  **derived from and conformance-tested against the vendored ext-apps schema**,
  not hand-maintained (D-182); the runtime stays Zod-free.
- **Vendored ext-apps schema** (new) ‚Äî the machine-readable
  `@modelcontextprotocol/ext-apps` schema, pinned into the repo by upstream SHA
  (RFC ¬ß10), from which `web/bridge`'s wire types are inferred and against which
  its outbound wire is conformance-tested. The View-side analogue of the ¬ß5.4
  `protocolcodec` seam / the ¬ß8.2 vendored Tasks schema.
- **Wire conformance test** (new) ‚Äî a `web/bridge` test that `.parse()`s the
  bridge's emitted `ui/` payloads against the vendored schema; the structural
  guard that turns silent wire drift into a failing build (the test that would
  have caught D-179).
- **Dockyard Apps extension** (new) ‚Äî a `ui/` notification Dockyard emits that is
  **not** in the MCP Apps schema (`task-progress`, `elicitation-response`),
  serving the Tasks√óApps surface (RFC ¬ß8, D-134). Functions only against a
  Dockyard-aware host (the inspector, Harbor); a stock host ignores it. Fenced in
  `dockyard-ext` (D-183).

## Implementation notes (deviations ‚Äî CLAUDE.md ¬ß4.3)

The wave shipped as specified with these clarifications, all recorded in D-182 /
D-183:

- **Schema referenced only by the test layer, not `protocol.ts`.** `dockyard-bridge`
  publishes as source (D-172); a schema import in the public type graph would
  force consumers to install `zod` + `@modelcontextprotocol/sdk` to type-check
  the bridge. So `protocol.ts` keeps clean hand-written types **pinned by the
  runtime `conformance.test.ts`** rather than derived by `z.infer` ‚Äî which is the
  *stronger* guard (a structural `extends` can't catch a renamed field; the
  round-trip `.parse()` does, and caught item A). The runtime stays Zod-free and
  consumers gain no dependency.
- **Bundle Zod-free guard = the static "schema referenced only by tests" smoke
  check**, not a build-and-grep ‚Äî Zod can only enter the bundle via a runtime
  import, which is statically forbidden.
- **devDep fallback, not pure file-vendor.** `schema.ts`'s closure pulls
  `@modelcontextprotocol/sdk/types.js`, so `zod` + the SDK are `web/bridge`
  devDeps; the ext-apps schema is vendored by SHA (`7d4434e`).
- **Inspector inbound-schema validation deferred** (V2-BACKLOG). Item 4 made the
  inspector host behaviourally faithful; `.parse()`-validating its inbound adds
  Zod to `web/inspector` for marginal value over the bridge's conformance test.
- **Item 6 (`resources/read` / `ping` / logging) deferred** to a V2-BACKLOG
  "additive-capability parity" entry ‚Äî additive features, not conformance bugs.
