## Summary

This wave brings Dockyard's MCP Apps wire output into line with how the
**vendored MCP Apps spec** (`docs/specifications/mcp-apps-2026-01-26.mdx`) and
the **official `@modelcontextprotocol/ext-apps` SDK** actually behave, closing
three friction points an upstream team hit when targeting Claude Desktop. The
items 1 and 6 from the original feedback (`.UI()` silent no-op, checkout-only
npm packages) already shipped in v1.5.0 (D-173, D-174) and are confirmed fixed
‚Äî this wave does not revisit them.

1. **`_meta.ui.domain` becomes an opt-in, host-supplied verbatim string;
   server-side auto-derivation is retired. (D-176 ‚Äî supersedes D-062/D-063;
   amends RFC ¬ß7.5.)** The spec defines `domain` as *"Host-dependent ‚Ä¶ Servers
   MUST consult host-specific documentation for the expected domain format ‚Ä¶
   If omitted, Host uses default sandbox origin"* ‚Äî the host **mints** the
   value; a server copies it verbatim or leaves it empty. Dockyard instead
   **auto-derived** Claude's `{hash}.claudemcpcontent.com` subdomain itself,
   which (a) re-implements a host's internal algorithm (the "host matrix"
   drift ¬ß6 forbids, in derivation-function clothing) and (b) is **rejected by
   Claude Desktop on a local connector** ‚Äî the exact error the upstream team
   hit. The fix: emit `App.Domain` verbatim, retire the synthesising Claude
   host profile, guard against `domain` on a stdio-only server, and document
   it as host-supplied + remote-only + opt-in.
2. **An opt-in flag additionally emits the deprecated flat
   `_meta["ui/resourceUri"]` key; the default stays nested-only. (D-177.)**
   The reference SDK emits **both** the nested `_meta.ui.resourceUri` and the
   flat `_meta["ui/resourceUri"]` "for backward compatibility." Dockyard emits
   nested-only by RFC ¬ß7.1 + brief 01 ¬ß2.3, and the 2026-01-26 spec marks the
   flat form deprecated ‚Äî so the default does **not** change. But because some
   hosts may still read the flat key, a server-level opt-in lets a developer
   emit both without us betting the RFC-compliant default on an **unproven**
   change (the feedback notes Claude Desktop did *not* render even with both
   keys present).
3. **The scaffold adopts the html-style `ui://<server>/<app>/index.html` URI
   convention, documented. (D-178.)** The reference app uses an html-style
   resource URI (`ui://deck-editor/index.html`); Dockyard scaffolds
   `ui://<server>/<app>` (no extension). The framework treats the URI as an
   opaque string, so this is a **convention + docs change only** ‚Äî no
   behaviour change ‚Äî but some hosts may key off the html-style path, and
   matching the reference removes one more diff.

## RFC anchor

- **RFC ¬ß7.1** ‚Äî what Dockyard registers: `_meta.ui` on tools (nested form
  only) and on the `resources/read` response (`csp`, `permissions`, `domain`,
  `prefersBorder`). Item 1 changes how `domain` is produced; item 2 adds an
  opt-out for the nested-only rule.
- **RFC ¬ß7.5** ‚Äî capability-driven graceful degradation, and the **two
  host-specific concerns**, the first of which is `_meta.ui.domain`
  auto-derivation. **Item 1 amends this section**: the spec makes `domain` a
  host-supplied verbatim value, so "Dockyard derives the dedicated iframe
  origin automatically, including host-specific signed forms" is wrong and
  becomes "Dockyard emits a developer-supplied, host-documented `domain`
  verbatim; it never synthesises a host's signed origin." The pluggable
  host-profile **seam** (P3 extensibility) is retained; the **synthesising
  Claude profile** is retired.
- **RFC ¬ß7.4** ‚Äî deny-by-default CSP / single-file bundles. Unchanged;
  referenced because `domain` and CSP share the `resources/read` `_meta.ui`
  choke point.
- **RFC ¬ß6** ‚Äî the contract-first `runtime/tool` builder; item 2's opt-in
  threads through the `_meta.ui` it already emits via `apps.ToolMetaFor`.

## Briefs informing this wave

- **brief 01 (MCP Apps spec teardown)** ‚Äî ¬ß2.2 the `ui://` resource + the

- **brief 04 (mcp-use DX teardown)** ‚Äî the "first-host" DX bar: a value that

## Brief findings incorporated

- **brief 01 ¬ß2.2 ‚Äî `domain` is read from the `resources/read` response, and
  it is host-determined.** The vendored spec text
  (`mcp-apps-2026-01-26.mdx:211-229`) is explicit: *host-dependent format*,
  *"Servers MUST consult host-specific documentation"*, *"If omitted, Host
  uses default sandbox origin."* Item 1 implements exactly this ‚Äî verbatim
  pass-through, default empty.
- **brief 01 ¬ß2.3 ‚Äî the nested `_meta.ui` form is canonical; the flat form is
  deprecated.** Item 2 keeps nested-only as the default and gates the flat key
  behind an explicit opt-in, so the RFC-compliant default is preserved.
- **brief 04 ‚Äî a default that lies is worse than no default.** The
  `HostProfile: "claude"` auto-derivation produced a confident-looking value
  that a real host rejects; retiring it + a stdio guard removes the trap.

## Findings I'm departing from (if any)

- **Departure from D-062/D-063 (settled decisions) and RFC ¬ß7.5.** Those
  decided that Dockyard auto-derives the dedicated origin server-side,
  including Claude's SHA-256 `claudemcpcontent.com` form, behind a host-profile
  derivation seam. The vendored spec and the reference SDK show this is the
  wrong model: `domain` is a host-minted, developer-copied verbatim value, not
  a framework-computed one ‚Äî and the derived value is rejected on local
  connectors. This wave **supersedes D-062/D-063** (D-176) and amends RFC ¬ß7.5.
  The host-profile **seam** is kept (it is sound extensibility per ¬ß4.4 and
  P3); only the **synthesising derivation** is retired. History is not
  rewritten: D-062/D-063 stand as the record of what was decided then; D-176 is
  the supersession.
- **Departure from the feedback's ¬ß3 recommendation ("emit both forms").** The
  feedback recommends emitting both keys; this wave makes it **opt-in** rather
  than the default, because (a) the change is explicitly unproven (Claude
  Desktop did not render even with both keys) and (b) emitting the deprecated
  flat key by default would contradict RFC ¬ß7.1 + the spec's deprecation and
  reverse the `protocolcodec` "never emit the flat form" golden/fuzz tests.
  Opt-in gives the downstream team a way to test it without a default
  regression.
- **Departure from the feedback's ¬ß5 recommendation (a `validate` check that a
  UI tool emits `_meta.ui` on the wire).** `validate`/`testgate` are static by
  design (D-082) ‚Äî they read the manifest + contracts, they do not build/run
  the server ‚Äî so a "present on the wire" assertion has no cheap home there
  (the same conclusion the v1.5 wave reached; it is already a V2-BACKLOG item).
  The static-feasible half (a `domain`-on-stdio guard) is implemented as a
  **runtime startup warning** in this wave; see the open question on a static
  variant.

## Goals

1. An App that sets `Domain` gets `_meta.ui.domain` **equal to that string
   verbatim** on every `resources/read`; an App that sets no `Domain` emits no
   `_meta.ui.domain` (the host's default per-conversation origin) ‚Äî matching
   the reference SDK and the spec. The synthesising Claude host profile no
   longer rewrites the value.
2. A server whose only transport is **stdio** that nonetheless has an App with
   a non-empty `Domain` produces a **loud startup warning** (`slog.Warn`)
   naming the App and explaining `domain` is remote-connector-only ‚Äî so the
   trap surfaces at `dockyard dev`/run instead of as a silent Claude-Desktop
   rejection.
3. A server-level **opt-in** makes every UI tool's `_meta.ui` additionally
   carry the flat `_meta["ui/resourceUri"]` key; the default (opt-in off) is
   unchanged nested-only output, still asserted by the existing
   `protocolcodec`/`apps` golden + fuzz tests.
4. `dockyard new` scaffolds an App `ui://` URI in the html-style
   `ui://<server>/<app>/index.html` convention, the templates follow, and the
   `attach-a-ui-resource` skill + docs document the convention. No framework
   behaviour depends on it (the URI is opaque).
5. RFC ¬ß7.5, `docs/decisions.md`, `docs/glossary.md`, the `apps.App` godoc, the
   `attach-a-ui-resource` skill, and the docs-site Apps guide all describe the
   new `domain` model; the ¬ß19 surfaces stay in sync in this PR.

## Non-goals

  against pengui, with a Claude-Desktop repro loop. It may yield a fix, a doc,
  or a confirmed host-side limitation ‚Äî but it is not a known Dockyard defect
  and does not gate this wave.
- **A static `validate` assertion that a UI tool emits `_meta.ui` on the
  wire.** Static-by-design `validate`/`testgate` (D-082) can't see runtime
  output; this remains the existing V2-BACKLOG item (an ephemeral-server run,
  the D-081 pattern).
- **Removing `App.HostProfile` / `App.ServerURL`.** Removing public fields is a
  breaking change (D-159). They are **deprecated** (godoc + no longer drive
  synthesis), not removed, keeping this a minor bump. (Lock deprecate-vs-remove
  in D-176 ‚Äî see open questions.)
- **Surfacing `domain` / `host_profile` as a manifest field.** The domain is
  set programmatically in `apps.Register`; adding a manifest field (to enable a
  static `validate` gate) is a coherent follow-up but expands scope ‚Äî recorded
  as an open question, not built here.

## Risks / open questions

- **Deprecate vs remove `App.HostProfile` / `App.ServerURL`.** Removing public
  fields is breaking (D-159 ‚Üí major). Recommendation: **deprecate** (godoc +
  no-op for derivation), keeping a minor bump; remove in a future major. Lock
  in D-176.
- **The flat-key opt-in knob's surface.** Leading candidate: a
  `server.Options` boolean (e.g. `EmitLegacyToolUIMeta`) that threads into
  `apps.ToolMetaFor` ‚Üí the `protocolcodec` encoder. Alternatives: a manifest
  `compat:` block (declarable + scaffoldable) or a per-builder `.UI()` option
  (too granular ‚Äî the concern is server-wide wire compat). Lock the shape in
  D-177 at implementation; the acceptance criteria are knob-agnostic.
- **A static `validate` variant of the domain-on-stdio guard.** The runtime
  `slog.Warn` catches it at run time. A *static* `validate` gate needs `domain`
  surfaced as a manifest field (it is Go-only today). Recorded as a follow-up
  (a manifest `apps[].domain` + a `checkAppDomainTransport` static check),
  not built here ‚Äî note in D-176.
- **URI-convention churn for existing projects.** The scaffold change does not
  touch existing projects (the URI is a free string); the docs explicitly say
  an existing `ui://<server>/<app>` URI keeps working. No migration required.
- **The host-profile seam after retiring the Claude derivation.** Keep the
  `HostProfile` interface + `RegisterHostProfile` (P3/¬ß4.4 extensibility) so a
  future *legitimate* host transform (one a host documents and blesses) has a
  home ‚Äî only the **synthesising** Claude profile is retired. Confirm the seam
  has ‚â•1 (generic verbatim) registered profile so the registry isn't empty.

## Glossary additions / changes

- **Dedicated origin** (rewrite) ‚Äî the stable, per-App sandboxed-iframe origin
  a host serves an App's HTML from (`_meta.ui.domain`). It is **host-minted and
  developer-supplied verbatim**: a developer copies the value the host
  documents for a verified remote deployment, or leaves it empty for the host's
  default per-conversation origin. Dockyard does **not** synthesise it. RFC
  ¬ß7.5. D-176 (supersedes the auto-derivation of D-062/D-063).
- **Domain label** (rewrite or retire) ‚Äî was "the host-agnostic label that
  feeds host-profile derivation." With derivation retired, `App.Domain` is the
  verbatim origin itself; update the entry to reflect verbatim passthrough (and
  cross-link Dedicated origin), or retire it if redundant.
