## Summary

This wave ships two items for v1.5:

1. **Tool→App `_meta.ui` link.** `tool.New[...].UI(appName).Register(srv)`
   silently drops the UI link (no `_meta.ui.resourceUri` on the tool def),
   so a real host renders the text fallback. Wire the builder to emit
   `_meta.ui` (RFC §7.1), fail loud on an unresolved `.UI()` name, add
   per-tool visibility control, and close the two gaps that hid it (a
   `dockyard test` guard + the skill's overstatement). **(D-173.)**
2. **Rename the npm packages to unscoped names.** `@dockyard/bridge` →
   `dockyard-bridge`, `@dockyard/ui` → `dockyard-ui`. The v1.4.0 release
   shipped binaries + a GitHub Release, but the npm publish 404'd:
   `@dockyard` is an org scope the maintainer cannot own. Unscoped names
   publish under the personal npm account; `@dockyard/inspector` (never
   published — the internal inspector frontend) is untouched. **(D-174 —
   supersedes D-172's naming.)**

### Item 1 detail

`tool.New[...].UI(appName).Register(srv)` **silently drops the UI link**:
the registered tool definition carries no `_meta.ui.resourceUri`, so a host
that renders MCP Apps (Claude Desktop) has nothing linking the tool result
to its `ui://` resource and falls back to the text rendering. The `ui://`
resource itself registers fine — only the tool→resource link is missing.
This wave makes the builder emit `_meta.ui` (RFC §7.1), fails loud on an
unresolved `.UI()` name, adds per-tool visibility control, and closes the
two secondary gaps that let it hide (a `dockyard test` guard, and the
skill's overstatement).

## RFC anchor

- RFC §7.1 — the tool↔App link (`_meta.ui.resourceUri` on the tool
  definition). The core fix emits it.
- RFC §7.5 — capability-driven graceful degradation: a host without the
  Apps extension ignores `_meta.ui`; the tool still works as a plain tool.
- RFC §6 — the contract-first `runtime/tool` builder this wave extends.

## Briefs informing this wave

- brief 01 (MCP Apps spec teardown) — §2.3 the nested `_meta.ui`
  `{resourceUri, visibility}` shape; §2.2 the `ui://` resource.
- brief 04 (mcp-use DX teardown) — the "first-run / first-host" DX bar; a
  silent no-op that costs a debugging session is exactly the friction the
  brief's lens flags.

## Brief findings incorporated

- **brief 01 §2.3 — the link lives in the tool definition's `_meta.ui`,
  nested `{resourceUri, visibility}`, never the deprecated flat form.** The
  fix routes through the already-shipped `apps.ToolMetaFor` →
  `internal/protocolcodec`, which emits exactly that nested form (P3).
- **brief 01 §2.3 — visibility is a tool-link property** (`model` / `app`),
  not an App property. Hence the `.UI()` visibility control lands on the
  builder, and `apps.App` is unchanged.
- **brief 04 — a gate that lies is worse than no gate.** `.UI("typo")` is a
  silent no-op and `dockyard validate` gave false confidence; this wave
  makes both fail loud.

## Findings I'm departing from (if any)

- **Deviation from this plan's own first draft (not a brief):** the draft
  proposed a `dockyard test` (`internal/testgate`) check asserting a
  UI-bearing tool emits `_meta.ui` at runtime. On implementation this proved
  both **infeasible cheaply** and **not the right guard**: `testgate` and
  `validate` are static by design (D-082 — they read the manifest +
  contracts, they do not build/run the user's server), and the upstream bug
  was a *framework* defect, not a user-project mistake — no user-facing gate
  could have caught it. The guard is therefore Dockyard's own regression
  test (the builder emits `_meta.ui`) + the fail-loud `Register`; the
  existing static `checkToolUIMappings` covers the manifest side. A runtime
  `_meta.ui`-present assertion would need an ephemeral-server run (the
  D-081 pattern) and is filed as a V2-BACKLOG follow-up, not built here.
- Otherwise none. The core fix implements documented intent: `ToolDef.Meta`'s
  own doc comment says "the Apps layer supplies `_meta.ui` here," and
  `UI()`'s doc said "the Apps layer consumes it to wire `_meta.ui`" — neither
  was ever wired.

## Goals

1. `tool.New[...].UI(appName).Register(srv)` produces a registered tool
   whose `tools/list` definition carries `_meta.ui.resourceUri` equal to the
   linked App's `ui://` URI (RFC §7.1) — so a real host renders the App.
2. A `.UI(name)` that names no registered App makes `Register` return a
   **typed error** (never a silent no-op).
3. A tool can declare its visibility (`model` / `app`) through the builder;
   the default (unspecified) is rendered as the spec default (model+app).
4. The canonical `analytics-widgets` template emits `_meta.ui` for its three
   widget tools with **no template change** (the fix is in the builder it
   already calls).
5. `dockyard test` catches a UI-bearing tool that does **not** emit
   `_meta.ui` (closing the false-confidence gap), and the
   `attach-a-ui-resource` skill documents the (now-true) `.UI()` behaviour +
   the apps-before-tools ordering.

## Non-goals

- Changing the `ui://` resource registration (`apps.Register` already works)
  or the `resources/read` `_meta` choke point.
- Auto-deriving the App URI from a naming convention (Option A) — rejected
  in favour of the server seam (D-173), which handles a custom `App.URI`.
- Enforcing `App.URI == ui://<server>/<name>` (the convention stays a
  convention; the seam makes it unnecessary).
- A breaking change to `apps.App` (no `Visibility` field added — visibility
  is a tool-link property).

## Acceptance criteria

### Core — the builder emits `_meta.ui`

- [ ] `runtime/server` gains a name→link seam: `type AppLink struct { URI
      string }`, `(*Server).RegisterAppLink(name string, link AppLink)
      error` (duplicate name → typed error), `(*Server).AppLinkByName(name
      string) (AppLink, bool)`.
- [ ] `apps.Register`, after `s.AddResource(...)` succeeds, records
      `s.RegisterAppLink(app.Name, server.AppLink{URI: app.URI})`.
- [ ] `tool.Builder.Register`, when `b.uiResource != ""`, looks up
      `s.AppLinkByName(b.uiResource)`; on a miss it returns a typed error
      naming the tool + the missing App and pointing at the
      apps-before-tools ordering; on a hit it sets `def.Meta =
      apps.ToolMetaFor(apps.ToolLink{ResourceURI: link.URI, Visibility:
      b.uiVisibility})`.
- [ ] The registered tool's `tools/list` definition carries
      `_meta.ui.resourceUri == <App URI>` and the **nested** form only
      (never the deprecated flat `ui/resourceUri` key) — the
      `TestRegisterAndDiscover` invariant, now proven through the builder.
- [ ] A host without the Apps extension still sees a working plain tool
      (the `_meta.ui` is ignored, RFC §7.5) — unchanged, asserted.

### Visibility control

- [ ] The builder can declare tool visibility (`model` / `app`) — e.g.
      `.UI(name, tool.VisibilityApp)` (the recommended variadic; the
      `.UIVisibility(...)` alternative is noted below). `tool.VisibilityModel`
      / `tool.VisibilityApp` consts mean a user need not import `runtime/apps`.
- [ ] No visibility declared → `_meta.ui` omits `visibility` (the host
      treats it as `[model, app]` — the spec default).
- [ ] `.UI(name, tool.VisibilityApp)` → `_meta.ui.visibility == ["app"]`
      (a UI-only action tool).

### Fail-loud

- [ ] `.UI("typo").Register(srv)` with no registered App named `typo`
      returns a typed error; a test asserts it.

### Secondary — close the gaps that hid it

- [ ] **The regression guard is Dockyard's own test, not a new user-facing
      gate** (deviation from the original plan — see below). The bug was a
      *framework* defect (the builder never emitted `_meta.ui`); a user's
      `dockyard validate` passed because their manifest was correct. So no
      user-facing gate "would have caught it" — only the framework's tests.
      The guard is the new `runtime/tool` builder test (`.UI().Register()`
      emits `_meta.ui.resourceUri`) + the fail-loud at `Register`. The
      existing static `checkToolUIMappings` (`internal/validate`) already
      covers the manifest side (tool `ui:` ↔ declared app).
- [ ] `skills/attach-a-ui-resource/SKILL.md`: document that `.UI(appName)`
      is now sufficient, the **apps-before-tools** ordering rule, and the
      visibility option.

## Public API surface

- `runtime/server`: `type AppLink struct { URI string }`,
  `(*Server).RegisterAppLink(name string, link AppLink) error`,
  `(*Server).AppLinkByName(name string) (AppLink, bool)` — additive.
- `runtime/tool`: `Builder.UI(resourceName string, visibility ...string)`
  gains the optional visibility variadic (backward-compatible — existing
  `.UI(name)` calls compile unchanged); `tool.VisibilityModel` /
  `tool.VisibilityApp` consts. `Register` gains a side-effect (emits
  `_meta.ui`) + a new failure mode (unresolved `.UI()` name).
- No change to `runtime/apps.App` / `ToolLink` / `ToolMetaFor`.

## Test plan

- **Unit (runtime/tool, the headline gap):** build a server, `apps.Register`
  an App, then `tool.New[...].UI(appName).Register(srv)`; connect a session,
  `tools/list`, assert `_meta.ui.resourceUri == appURI` and no flat key.
  `.UI("typo").Register` → typed error. `.UI(name, VisibilityApp)` →
  `visibility == ["app"]`; `.UI(name)` → no `visibility` key.
- **Unit (runtime/server):** `RegisterAppLink`/`AppLinkByName` round-trip;
  duplicate name → error.
- **Unit (runtime/apps):** `Register` records the link
  (`AppLinkByName(app.Name)` resolves to `app.URI`).
- **Integration (`test/integration`, real server):** scaffold/registration
  mirroring the `analytics-widgets` shape (App registered, then three
  `.UI(appName)` tools) → `tools/list` shows `_meta.ui` on all three; a
  registration that wires a tool before its App errors.
- **Unit (internal/testgate):** the new capability check fails a project
  whose UI-bearing tool emits no `_meta.ui`, passes one that does.

## Risks / open questions

- **Ordering rule (apps before tools).** Option B requires an App to be
  registered before a tool that links it. The template and the upstream
  project already satisfy it (`registerApp` before `registerTools`), and the
  fail-loud error names the fix. **Open question:** should a deferred/
  two-pass resolution ever be supported (register tools, resolve links at
  serve time)? Deferred — the ordering rule is simple and the error is
  clear; record the decision either way in D-173.
- **Visibility API shape.** Recommended: a variadic on `.UI(name,
  vis...)` — one call, backward-compatible. Alternative considered: a
  separate `.UIVisibility(...)` method (more self-documenting, but a second
  call + an ordering concern). Lock the choice at implementation time and
  record it in D-173.
- **`tool` importing `apps`.** Verified no import cycle (apps→server,
  tool→server; tool→apps adds none). If a future refactor risks one, the
  meta-encoding entry point can move behind a `server`-level seam instead.
- **`dockyard test` guard scope.** It must build/connect the server to read
  `tools/list`; `internal/testgate`'s capability category already does this,
  so the check lands there, not in static `validate`.

## Glossary additions

- **tool→App link** — the `_meta.ui.resourceUri` (+ optional `visibility`)
  on a tool definition that tells a host which `ui://` App resource renders
  the tool's result (RFC §7.1). Built by `apps.ToolMetaFor`; wired by the
  `runtime/tool` builder's `.UI()`. (Add to `docs/glossary.md` if absent.)
