# Scaffold a Dockyard MCP server

Dockyard ships one CLI binary — `dockyard` — and one scaffold verb:
`dockyard new`. There are two paths:

1. **Blank server** (the first-class path): `dockyard new <name>` — one
   manifest, one example contract-first tool, the generated artifacts, a
   runnable `main.go`, and a contract test.
2. **Product-pattern template**: `dockyard new <name> --template <slug>` —
   one of the V1 templates that ships a real product path end-to-end.

Pick the blank path when you want to understand the framework. Pick a
template when you want a worked example you can adapt.

## Pre-flight check

```bash
dockyard --help        # verify the CLI is on PATH
dockyard new --help    # confirm the flags below
```

If `dockyard` is not on PATH, build it from the Dockyard checkout:

```bash
cd /path/to/dockyard
make build
export PATH="$PWD/bin:$PATH"   # bin/dockyard
```

## Blank server — the first-class path

```bash
dockyard new my-server
cd my-server
go mod tidy             # populate go.sum from the resolved replace target
go test ./...           # the scaffolded contract test
go run .                # serves over stdio on the current terminal
```

`go mod tidy` is required once after a pre-publish scaffold — the
generated `go.mod` carries a `replace` directive but no `go.sum`, so the
first `go test` / `go run` would otherwise fail to resolve transitive
deps.

The scaffold produces:

```text
my-server/
├── README.md
├── dockyard.app.yaml          # the manifest — RFC §4.2
├── go.mod
├── greet.go                   # registerTools + the greet handler
├── greet_test.go              # contract test
├── main.go                    # stdio | http serve
└── internal/
    └── contracts/
        └── contracts.go       # GreetInput / GreetOutput — the source of truth
```

The manifest's `tools[]` block points at the Go types (`internal/contracts.
GreetInput` / `GreetOutput`); the JSON Schema is generated by
`dockyard generate`. The blank server has no UI — no `apps[]` block.

## Template — a worked product path

The two V1 templates:

- **`analytics-widgets`** — three contract-first widget tools
  (`create_chart`, `create_table`, `create_metric_card`) rendered inline by
  one Svelte App. The canonical read-side example.
- **`approval-flows`** — two task-augmented tools (`request_approval`,
  `propose_with_edits`) rendered as a human-in-the-loop card / form by one
  Svelte App; drives the `input_required` round-trip. The canonical
  write-side example.

```bash
dockyard new my-widgets --template analytics-widgets
cd my-widgets
ls   # main.go, dockyard.app.yaml, web/, internal/, pkg/contracts, …
```

A template scaffold drops in the Svelte App under `web/`, the typed
contracts under `internal/contracts/`, and the handlers under
`internal/handlers/`. The manifest's `apps[]` block declares
`display_modes: [inline]` and a deny-by-default CSP.

## Pre-publish: `--dockyard-path`

Until Dockyard is published to a module registry, a scaffolded project's
`go.mod` needs a `replace` directive pointing at the local Dockyard
checkout. The CLI's hidden `--dockyard-path` flag writes the replace for
you and also wires the project's `web/package.json` workspace to the
in-repo `@dockyard/bridge` and `@dockyard/ui` packages:

```bash
dockyard new my-widgets \
  --template analytics-widgets \
  --dockyard-path /Users/me/code/dockyard
```

The flag is hidden because a released `dockyard` will not need it (D-080).
Skip it once Dockyard is published.

## Other flags

- `--dir <path>` — parent directory for the project (default: cwd).
- `--module <go-module>` — Go module path for `go.mod` (default:
  `example.com/<name>`).

## Verify the scaffold

For any scaffold path:

```bash
cd <project>
go mod tidy             # only needed once, after a pre-publish scaffold
dockyard generate       # template scaffolds need this once before validate (blank scaffolds ship them pre-generated)
dockyard validate       # quality gates — should report 0 blockers
go test ./...
```

If `dockyard validate` reports stale codegen on a fresh scaffold, that is
a Dockyard bug; file it.

## Common pitfalls

- **Existing directory** — `dockyard new` refuses to overwrite an existing
  project. Choose a different name or remove the directory first.
- **Invalid name** — names must be lowercase letters, digits and hyphens,
  no leading/trailing hyphen. The CLI prints the rule when the name is
  invalid.
- **Missing template** — `--template unknown` exits non-zero with the list
  of registered templates; see `dockyard new --help`.
